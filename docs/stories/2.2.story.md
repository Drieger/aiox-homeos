## Story

**As a** usuário,
**I want** visualizar a escala tipográfica e a paleta de cores do HomeOS,
**so that** eu tenha uma referência visual dos tokens de design do sistema.

## Complexity

**Estimativa:** S-M (Small-Medium) — componentes de display estático, mas requer acesso ao sistema de temas (useTheme) e mapeamento correto dos CSS variables do shadcn/ui.

## Scope

**IN:**
- Seção "Tipografia" com H1, H2, H3, H4, body, small, muted, code — com nome do token e exemplo
- Seção "Cores" com tokens CSS do tema shadcn/ui (light + dark side-by-side)
- Cada swatch: nome do token, CSS variable (`--background`, etc.) e preview colorido
- Ambas as seções respondem ao toggle de tema global
- 1 teste de componente (TDD)

**OUT:**
- Editor de tokens ou customização de cores
- Exportação de tokens
- Animações ou transições além do theme switch nativo

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Lista de CSS variables do shadcn/ui incompleta | Média | Baixo | Mapear as variáveis diretamente do `globals.css` gerado pelo shadcn/ui init |
| `useTheme` causa hydration mismatch | Baixa | Médio | Usar `mounted` state para renderizar swatches apenas client-side |

4. Ambas as seções respondem ao toggle de tema

### Cores — CSS variables do shadcn/ui

O shadcn/ui define variáveis em `globals.css` (gerado pelo `npx shadcn@latest init`) no formato `--background: oklch(...)` ou `hsl(...)`. Para mostrar light + dark simultaneamente, renderizar dois grupos com classes `.light` / `.dark` explícitas ou usar `data-theme` attributes. [tech-stack.md — next-themes]

### Client Component obrigatório

`useTheme` do `next-themes` requer `"use client"`. O componente de cores deve ser Client Component. Componente de tipografia pode ser Server Component (sem estado). [coding-standards.md#2-padrões-de-componentes-react]

### Hydration SSR-safe

```typescript
const [mounted, setMounted] = useState(false)
useEffect(() => setMounted(true), [])
if (!mounted) return null // evita mismatch
```

[tech-stack.md — next-themes SSR-safe com `suppressHydrationWarning`]

### Bug Identificado e Corrigido (Refactoring Direto)

**Bug:** Seção "Dark" exibia fundo branco — texto "Dark" (branco via `--foreground`) era invisível, bordas eram imperceptíveis.

**Causa Raiz:** `ThemeGroup` não tinha `bg-background` no container. As variáveis CSS do `.dark` eram herdadas corretamente (texto ficava branco via `--foreground: oklch(0.985 0 0)`), mas o `background-color` da div nunca era definido — ficava transparente, exibindo o fundo branco da página. Bordas `border-border` em dark = `oklch(1 0 0 / 10%)` (branco 10%) eram invisíveis sobre branco.

**Correção aplicada diretamente (Quinn):** Adicionado `bg-background` ao className do `ThemeGroup`:
- **Arquivo:** `homeos/src/components/design-system/colors-section.tsx`
- **Mudança:** `className="flex-1 rounded-lg border border-border p-4 ${theme}"` → `className="flex-1 rounded-lg border border-border bg-background p-4 ${theme}"`
- **Resultado:** Grupo Dark → `--background: oklch(0.145 0 0)` → fundo escuro aplicado. Texto branco e bordas sutis ficam visíveis corretamente.

**Validações pós-fix:** 26/26 testes ✅ · lint ✅ · build ✅

### Bug Identificado e Corrigido (Refactoring Direto)

**Bug:** Labels dos tokens ("background", "foreground", "card", ...) apareciam em preto no grupo Dark, ficando invisíveis sobre o fundo escuro.

**Causa Raiz:** `TokenRow` linha 32: `<span className="text-sm font-medium">` sem classe de cor. Cor padrão do browser = preto. **Correção aplicada (Quinn):** Adicionado `text-foreground` ao span do label:
- **Arquivo:** `homeos/src/components/design-system/colors-section.tsx`
- **Mudança:** linha 32: `text-sm font-medium` → `text-sm font-medium text-foreground`
- **Resultado:** Em dark mode, `--foreground: oklch(0.985 0 0)` = branco → labels legíveis sobre fundo escuro.

**Validações:** 26/26 testes ✅ · lint ✅

### Nota de Padrão Sistêmico

Este bug e o anterior (`bg-background` faltando) têm a mesma raiz: Tailwind sem `text-foreground` explícito em componentes que vão ser renderizados em contexto de tema variável. Em qualquer container com classe de tema (`.light` / `.dark`), **todos** os elementos de texto precisam de `text-foreground` ou `text-muted-foreground` explícitos — nunca depender da cor padrão do browser. Registrar como gotcha para Stories 2.3 e 2.4.
