## Story

**As a** usuário do HomeOS,
**I want** escolher entre tema claro, escuro ou automático (seguir o sistema),
**so that** minha preferência visual seja respeitada e persistida entre sessões.

## Complexity

**Estimativa:** S (Small) — infraestrutura `next-themes` já configurada; trabalho exclusivamente de UI (componente toggle + integração no layout).

## Scope

**IN:**
- Componente `ThemeToggle` com 3 opções: `system`, `light`, `dark`
- Cycling ao clicar: system → light → dark → system
- Ícones: Monitor (system), Sun (light), Moon (dark) via `lucide-react`
- Integração no header mobile (`(app)/layout.tsx`)
- Integração no rodapé da sidebar desktop (`app-sidebar.tsx`)
- Persistência automática via `localStorage` (gerenciada pelo `next-themes`)
- 1 arquivo de testes cobrindo os 3 estados e a lógica de cycling

**OUT:**
- Dropdown/popover com opções visíveis simultaneamente
- Preferência persistida em backend ou banco de dados
- Toggle de tema por página ou por seção
- Animação de transição ao trocar tema (já desabilitada pelo `disableTransitionOnChange`)

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Flash de hidratação (FOUC) | Média | Médio | Usar estado `mounted` para renderizar placeholder até hidratação completa |
| `theme` undefined antes da hidratação | Alta | Baixo | Fallback para `system` quando `theme` é undefined |
| Sidebar colapsada (w-16) sem espaço para label | N/A | N/A | Componente usa apenas ícone — sem texto visível |

## Acceptance Criteria

1. Header mobile exibe botão de toggle de tema acessível
2. Sidebar desktop exibe botão de toggle de tema no rodapé
3. Ao clicar, o tema é aplicado imediatamente sem reload de página
4. A sequência de cycling é: system → light → dark → system
5. A preferência é persistida em `localStorage` — ao recarregar a página, o tema selecionado é restaurado
6. A opção "system" segue automaticamente a preferência do sistema operacional
7. Sem FOUC (Flash of Unstyled Content) na hidratação inicial

## Dev Notes

### next-themes já configurado

`ThemeProvider` em `src/components/providers.tsx` usa:
```typescript
<ThemeProvider
  attribute="class"
  defaultTheme="system"
  enableSystem={true}
  disableTransitionOnChange
>
```

`useTheme()` retorna: `{ theme, setTheme, resolvedTheme, systemTheme }`.
- `theme`: valor explícito salvo (`"system"` | `"light"` | `"dark"` | `undefined`)
- `resolvedTheme`: tema efetivo (`"light"` | `"dark"`) — útil para exibir ícone correto no modo system

### Padrão mounted para evitar FOUC

```typescript
const [mounted, setMounted] = useState(false)
useEffect(() => setMounted(true), [])
if (!mounted) return <div className="size-8 rounded-md" />
```

Necessário porque `next-themes` não conhece o tema no servidor.

### Persistência

`next-themes` persiste automaticamente em `localStorage` com a key `theme`.
Nenhuma lógica adicional necessária.

### Import path

```typescript
import { ThemeToggle } from "@/components/layout/theme-toggle"
```

### Gotcha: @base-ui/react não usa asChild

Se precisar de triggers estilizados com @base-ui, usar `className={cn(buttonVariants({variant}))}` ao invés de `asChild`. [Lição aprendida Story 2.4]

Para este componente, usar `<button>` nativo estilizado — sem necessidade de shadcn Button.

### Testing — mock de next-themes

```typescript
vi.mock("next-themes", () => ({
  useTheme: () => ({
    theme: mockTheme,
    setTheme: mockSetTheme,
  }),
}))
```
