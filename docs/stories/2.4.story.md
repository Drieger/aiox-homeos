## Story

**As a** usuário,
**I want** ver e disparar os componentes de overlay e feedback do Design System,
**so that** eu possa verificar o comportamento de modais, sheets e notificações.

## Complexity

**Estimativa:** M (Medium) — componentes com comportamento de abertura/fechamento, integração com Toaster provider no root layout.

## Scope

**IN:**
- Seção "Overlays": `Dialog`, `Sheet`, `Tooltip`, `Popover`, `DropdownMenu`
- Seção "Feedback": `Toast` (com botão trigger), `Alert` (variantes), `Skeleton`, `Progress`
- Cada overlay abre e fecha via interação com trigger
- Toast disparado por botão e desaparece após timeout
- Verificar se `Toaster` está no root layout; adicionar se ausente
- 1 teste de componente (TDD): triggers de Dialog e Sheet presentes e clicáveis

**OUT:**
- Formulários dentro de Dialog
- Implementação de Toast como sistema de notificação global da aplicação
- Customização de Progress com dados reais

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Toaster não está no root layout | Média | Alto | Verificar `src/components/providers.tsx` e `src/app/layout.tsx`; adicionar se necessário |

| shadcn/ui usa Sonner para toast (não o `useToast` legado) | Baixa | Baixo | Verificar qual está instalado; shadcn/ui atual usa Sonner por padrão |

## Dev Notes

### Dependência

Esta story **depende de Story 2.1** (Design System Page Scaffold). A seção "Overlays" na rota `/design-system` deve existir antes de implementar o showcase. Implementar após Story 2.1 estar Done.

### Toaster Provider (Sonner)

shadcn/ui atual usa Sonner como provider de toast. O `<Toaster />` deve estar no root layout:

```typescript
import { Toaster } from "@/components/ui/sonner"
// Em src/components/providers.tsx ou diretamente em src/app/layout.tsx
```

Para disparar: `import { toast } from "sonner"` → `toast("Mensagem")`. [source-tree.md — `src/components/providers.tsx`]

### Dialog (Radix UI / shadcn)

```typescript
import { Dialog, DialogContent, DialogHeader, DialogTitle, DialogTrigger } from "@/components/ui/dialog"
// IMPORTANTE: DialogTitle é obrigatório para acessibilidade (Radix UI Aria requirement)
```

Overlays Radix usam portals — renderizam em `document.body`. Testing Library detecta portals nativamente. [tech-stack.md — Radix UI com WCAG AA]

### Tooltip requer TooltipProvider

```typescript
import { Tooltip, TooltipContent, TooltipProvider, TooltipTrigger } from "@/components/ui/tooltip"

<TooltipProvider>
  <Tooltip>
    <TooltipTrigger>Hover aqui</TooltipTrigger>
    <TooltipContent>Conteúdo do tooltip</TooltipContent>
  </Tooltip>
</TooltipProvider>
```

### Alert variantes

```typescript
import { Alert, AlertDescription, AlertTitle } from "@/components/ui/alert"
// variant="default|destructive"
```

### Gotcha: text-foreground em containers de tema (Story 2.2)

Em qualquer container com classe de tema explícita (`.light` / `.dark`), **todos** os elementos de texto precisam de `text-foreground` ou `text-muted-foreground` explícitos. Aplicável especialmente a `Alert` e labels dentro de overlays que possam ser renderizados com tema variável.

```typescript
// ✅ Correto
<span className="text-sm text-foreground">Texto legível em qualquer tema</span>

// ❌ Errado — invisível em dark mode
<span className="text-sm">Texto</span>
```

[Lição aprendida Story 2.2 — QA re-review `77d417c`]

### Client Component obrigatório

Todos os overlays e toast requerem `"use client"` (estado de abertura gerenciado por Radix UI internamente). [coding-standards.md#2-padrões-de-componentes-react]

### Testing — Portals Radix

Para testar Dialog/Sheet com Testing Library:

```typescript
import { render, screen } from "@testing-library/react"
import userEvent from "@testing-library/user-event"

it("trigger do Dialog está presente", async () => {
  render(<OverlaysSection />)
  const trigger = screen.getByRole("button", { name: /abrir dialog/i })
  expect(trigger).toBeInTheDocument()
})
```
